A few months back, while randomly browsing old Hacker News submissions, I came across a rather interesting submission from Simon Willison on ChatGPT Containers. What immediately caught my attention was not the post itself but one of the comment threads on HN:

xnx: How much compute do you get in these containers? Could I have it run whisper on an mp3 it downloads?

simonw: That might work! You would have to figure out how to get Whisper working in there but I'm sure that's possible with a bit of creativity concerning uploading files and maybe running a build with the available C compiler. It appears to have 4GB of RAM and 56 (!?) CPU cores https://chatgpt.com/share/6977e1f8-0f94-8006-9973-e9fab6d24418

56 LPs?? 4GB of RAM??! One of the users in the thread, named tintor, who appears to be ex-OpenAI (O1 reasoning model and code interpreter), briefly mentioned that the cores are shared with other containers. However, even with oversubscription, the number seemed baffling. So, I decided to further investigate this. In this blog post, we will dive into what their container environment looks like.

Before we begin, note that all of my chat prompts and their subsequent containerized executions were performed on ChatGPT's paid plan - ChatGPT Plus. They were running GPT 5.5 Extended Thinking.

This is strong evidence that ChatGPT uses gVisor as a sandboxing environment for code execution. To better understand gVisor, I encourage checking out Emma Haruka Iwao's 2019 talk and Ye Lin's recent blog post. Andrea and Remzi Arpaci-Dusseau (of OSTEP fame) have also co-authored a paper on it. Here is what the gVisor team has to say about their product:

gVisor is a container security solution. ….. An open source project written in Go, gVisor was released in May 2018 by Google under the Apache 2.0 license. It runs on Linux and integrates with all popular container management software, such as Docker, Podman, or Kubernetes. At its core, gVisor is an application kernel that implements a substantial portion of the Linux system call interface. This means gVisor sits between a container and the Linux kernel and plays both roles: from the container's perspective, gVisor acts as a kernel, but from Linux's perspective, gVisor is just a regular application. That means the container can no longer directly interface with the Linux kernel. This is a massive reduction in attack surface.
gVisor manages to pull the above trick with the help of two components:
  • Sentry is the component that runs the containerized application. It intercepts every system call that the application makes and reimplements it in Go. As part of this, it may decide to do one or more system calls to the host Linux kernel. However, it's heavily restricted with a strict seccomp filter (that's why system calls like open, socket, or exec are not allowed).
  • Gofer is a component that runs outside the container and is responsible for filesystem operations. The sentry may make I/O requests to the gofer. The gofer will independently validate them, then perform these I/O operations on the container's behalf (that's how the container can read files from the host filesystem, even though open is not allowed from the sentry).

Circling back to the above dmesg output, you might notice that the logs appear somewhat nonsensical. Why is that? By default, dmesg is used to print the messages in the kernel ring buffer. The modern interface to read this message buffer involves /dev/kmsg. The older interface involved reading from syslog. However, if you ask ChatGPT to perform the following command in its container: ls -l /dev/kmsg, you will get an "ls: cannot access '/dev/kmsg': No such file or directory" error. This is because /dev/kmsg is not present in their gVisor sandbox, and the command is likely to fall back to the older syslog method.

So what happens next? Observe the 103 syscall number in gVisor's linux64.go. 

103: syscalls.PartiallySupported("syslog", Syslog, "Outputs a dummy message for security reasons.", nil),

This is Sentry's syscall table that maps gVisor's syscall API to Linux's corresponding syscall numbers. The 103 syscall on Linux is syslog, which is getting wired to gVisor's Syslog handler. The code for the handler is at sys_syslog.go, which in turn invokes syslog.go's Log() function, which returns a dummy copy of the syslog.

The gVisor code corresponding to this seems to originate from pkg/cpuid/cpuid_amd64.go, particularly the WriteCPUInfoTo call: 

// WriteCPUInfoTo is to generate a section of one cpu in /proc/cpuinfo. This is
// a minimal /proc/cpuinfo, it is missing some fields like "microcode" that are
// not always printed in Linux. Several fields are simply made up.

From the code, the model name and stepping are hardcoded to "unknown". Furthermore, gVisor pretends that all CPUs are in the same socket and the physical id is always set to 0. The presence of fpu, fpu_exception, and wp (write protect) is also hardcoded to return "yes". Address sizes are not read from hardware either; they are predetermined to be 46 bits physical, 48 bits virtual. Perhaps the most interesting one is cache size, which is completely made up at 8192 KB with the comment: 

// Pretend the CPU has 8192 KB of cache. Note that real /proc/cpuinfo exposes total L3 cache
// size on Intel and per-core L2 cache size on AMD (as of Linux 6.1.0), so the value of this
// field is not really important in practice. Any value that is chosen here will be wrong
// by an order of magnitude on a significant chunk of x86 machines.
// 8192 KB is selected because it is a reasonable size that will be effectively usable on
// lightly loaded machines - most machines have 1-4MB of L3 cache per core.

But some fields contain real CPUID data. The vendor id, cpu family, model, and flags are all parsed using CPUID. CPUID is an instruction that is used to query processor details. Intel concisely describes this in their Architecture Software Developer Manuals:

This instruction returns the family, model, and (for some processors) a brand string for the processor that executes the instruction. It also indicates the features that are present in the processor and gives information about the processor's caches and TLB. .....
To obtain processor identification information, a source operand value is placed in the EAX register to select the type of information to be returned. When the CPUID instruction is executed, selected information is returned in the EAX, EBX, ECX, and EDX registers.

To fetch information from CPUID, WriteCPUInfoTo invokes the following call: 

ax, _, _, _ := fs.query(featureInfo)

Inside native_amd64.go, the featureInfo constant is defined as: 

featureInfo                   cpuidFunction = 0x1               // Returns basic feature bits and processor signature.

fs.query() is a thin wrapper that forwards the call to Native.Query(): 

// Query executes CPUID natively.
//
// This implements Function.
//
//go:nosplit
func (*Native) Query(in In) Out {
	if int(in.Eax) < len(allowedBasicFunctions) && allowedBasicFunctions[in.Eax] {
		return native(in)
	} else if in.Eax >= uint32(extendedStart) {
		if l := int(in.Eax - uint32(extendedStart)); l < len(allowedExtendedFunctions) && allowedExtendedFunctions[l] {
			return native(in)
		}
	}
	return Out{} // All zeros.
}

In the above code, Native.Query() checks whether the requested CPUID is in an allowlist. If allowed, it calls native(), whose implementation lives in the assembly file native_amd64.s. In this scenario, the source operand value 0x1 is put into the EAX register, and when the CPUID instruction is executed, the selected information is returned in the EAX, EBX, ECX, and EDX registers. 

TEXT ·native(SB),NOSPLIT|NOFRAME,$0-24
	MOVL arg_Eax+0(FP), AX
	MOVL arg_Ecx+4(FP), CX
	CPUID
	MOVL AX, ret_Eax+8(FP)
	MOVL BX, ret_Ebx+12(FP)
	MOVL CX, ret_Ecx+16(FP)
	MOVL DX, ret_Edx+20(FP)
	RET

The important thing to note here is that gVisor is not fabricating this data, unlike the others. It is running the real CPUID instruction. As for its runtime environment, a hypothesis can be made that OpenAI's gVisor is running inside a containerized environment managed by a hypervisor on a cloud infrastructure. Notice how the hypervisor flag is set in /proc/cpuinfo. Microsoft describes this approach to hypervisor discovery in its docs:

Before using any hypervisor interfaces, software should first determine whether it's running within a virtualized environment. On x64 platforms that conform to this specification, this is done by executing the CPUID instruction with an input (EAX) value of 1. Upon execution, code should check bit 31 of register ECX (the “hypervisor present bit”). If this bit is set, a hypervisor is present. In a non-virtualized environment, the bit will be clear.

In such a scenario, the hypervisor would intercept CPUID and modify its results. Hypervisors "can" present a fake CPU model - Chips and Cheese wrote a blog post describing this back in 2022. Therefore, it is hard to say whether the underlying hardware of the host might be Intel-based or not. The CPU family 6 and model 79 themselves seem to correspond to the Broadwell CPU family. It is rather old, being the fifth generation of the Intel Core processor from 2014. A handy way to find out the architecture correlated with a CPU family is by peeking at the constants in Linux's source code at linux/arch/x86/include/asm/intel-family.h - "#define INTEL_BROADWELL_X IFM(6, 0x4F)"

There is something fascinating we can do here. CPUID is an unprivileged instruction, not a syscall, and it executes in ring 3 without trapping the kernel. Therefore, we can create and execute a small C program that invokes the CPUID instruction and parses various processor details out of it.

Here is the program that I created with the help of Claude - the underlying comments and explanations are my own. 

#include <stdio.h>
#include <stdint.h>

static inline void cpuid_count(uint32_t leaf, uint32_t subleaf,
                               uint32_t *eax, uint32_t *ebx,
                               uint32_t *ecx, uint32_t *edx) {
   /* 
       Helper to execute a CPUID instruction.
      
       See https://wiki.osdev.org/Inline_Assembly for details on __asm__
       This is inspired by https://wiki.osdev.org/CPUID#Using_CPUID_from_GCC

       In the below code, a, b, c, d are termed as "constraints" and they tell where the C variable
       should be mapped - in this case, to the registers eax, ebx, ecx, and edx.
       For a complete list, see https://gcc.gnu.org/onlinedocs/gcc/Machine-Constraints.html
      
       As mentioned earlier in the blog, CPUID takes no parameters.
       Instead, source operand value is placed in the EAX register - the "leaf".
       Sometimes, the ECX register is also used to specify a "subleaf" for more detailed queries.
   */
   __asm__ __volatile__ (
       "cpuid"
       : "=a"(*eax), "=b"(*ebx), "=c"(*ecx), "=d"(*edx)
       : "a"(leaf), "c"(subleaf)
   );
}

int main(void) {
   uint32_t eax, ebx, ecx, edx;

   /* Wiki page is a great reference for CPUID information
       https://en.wikipedia.org/wiki/CPUID
   */

   /* Max leaf number that the processor supports is in EAX */
   cpuid_count(0x0, 0, &eax, &ebx, &ecx, &edx);
   uint32_t max_leaf = eax;
   printf("Max basic CPUID leaf: 0x%x\n", max_leaf);

   /* CPU's manufacturer ID string (12 char ASCII) lives in EBX, EDX, ECX (in that order). */
   char vendor[13];
   *(uint32_t*)(vendor + 0) = ebx;
   *(uint32_t*)(vendor + 4) = edx;
   *(uint32_t*)(vendor + 8) = ecx;
   vendor[12] = '\0';
   printf("Vendor: %s\n", vendor);

   /* Leaf 0x1:
       Claude's intention behind drafting the below function was not fully accurate!

       Here's why - Wikipedia tells me:
           1. EBX[23:16]:
               "Maximum number of addressable IDs for logical processors in this physical package"
               What this means is - every LP needs a unique number so that system can address it.
               This number is its APIC ID. By "addressable", it means a way to reach the LP by this ID number.
               So, the Wiki sentence probably equates to the biggest number of APIC IDs the chip has set aside.

               However, the bits for APIC ID seem to be allocated in powers of 2.
               So, if the chip has 6 LPs, it would need to allocate 8 APIC IDs (0-7) to be able to address them.
               So, the value in EBX[23:16] would be 8, not 6. The two reserved APIC IDs would just go unused.
               Therefore, we can estimate an upper bound on the number of LPs, not the exact number.
          
           2. EDX[28]: HTT flag
               This bit used to be about Hyper-Threading.
               See this blog: https://www.os2museum.com/wp/htt-means-hyper-threading-right/
               It says that: "As of October 2017, Intel now says: HTT:
               Max APIC IDs reserved field is Valid. A value of 0 for HTT indicates
               there is only a single logical processor in the package and software
               should assume only a single APIC ID is reserved. A value of 1 for HTT
               indicates the value in CPUID.1.EBX[23:16] (the Maximum number of addressable
               IDs for logical processors in this package) is valid for the package."
   */
   cpuid_count(0x1, 0, &eax, &ebx, &ecx, &edx);
   uint32_t logical_per_pkg = (ebx >> 16) & 0xFF;
   int has_htt = (edx >> 28) & 1;
   printf("\nLeaf 0x1:\n");
   printf("  Max logical processors per package: %u\n", logical_per_pkg);
   printf("  HTT flag: %d\n", has_htt);

   /* Leaf 0x4: Deterministic Cache Parameters */
   printf("\nLeaf 0x4 (Cache Topology):\n");
   for (int i = 0; i < 16; i++) {
       cpuid_count(0x4, i, &eax, &ebx, &ecx, &edx);
       /* 5 bits for cache type
           0 is no more caches, 1 is data cache, 2 is instruction cache,
           3 is unified cache, 4-31 are reserved.
       */
       int cache_type = eax & 0x1F;
       if (cache_type == 0) break;  /* no more caches */
       int cache_level = (eax >> 5) & 0x7; /* 3 bits for cache level (1,2,3) */
       /* Same as the APIC ID explanation from above
           Wiki says: "Maximum number of addressable IDs for logical processors sharing this cache, minus 1"
           So, after +1, this is the maximum number of addressable APIC IDs that tells us
           the upper bound on the number of LPs sharing this one cache instance.
           This is not a guarantee on the exact count either.
       */
       int max_threads_sharing = ((eax >> 14) & 0xFFF) + 1;
       /*
           Wiki says: "Maximum number of addressable IDs for processor cores in physical package, minus 1"
           So, after +1, this is the maximum number of addressable APIC IDs that tells us
           the upper bound on the count of "physical cores" in the package reported by this cache.
       */
       int max_cores_in_pkg = ((eax >> 26) & 0x3F) + 1;
       int line_size = (ebx & 0xFFF) + 1;
       /* Hierarchically cache is like: 
           Cache contains "sets" - groups of cache lines.
           Each "set" contains "ways" - the number of lines that can be stored in that set.
           Each "way" contains a cache line of "line_size" bytes.
           Partitions seem to be usually set to 1 - I do not fully understand them!
           So, total cache size = ways * line_size * sets * partitions.
       */
       int partitions = ((ebx >> 12) & 0x3FF) + 1;
       int ways = ((ebx >> 22) & 0x3FF) + 1;
       int sets = ecx + 1;
       int cache_size = ways * partitions * line_size * sets;
       const char *types[] = {"Null","Data","Instruction","Unified"};
       printf("  L%d %s: %d KB, %d-way, shared by %d threads, "
              "max cores in pkg: %d\n",
              cache_level,
              cache_type < 4 ? types[cache_type] : "???",
              cache_size / 1024, ways,
              max_threads_sharing, max_cores_in_pkg);
   }

   /* Leaf 0xB: Extended Topology Enumeration */
   printf("\nLeaf 0xB (Extended Topology):\n");
   if (max_leaf >= 0xB) {
       for (int level = 0; level < 8; level++) {
           cpuid_count(0xB, level, &eax, &ebx, &ecx, &edx);
           int level_type = (ecx >> 8) & 0xFF; /* 1 is SMT, 2 is Core */
           if (level_type == 0) break;  /* 0 is invalid level */
           /* It helps decode an x2APIC ID into (thread, core, package) coordinates
           for the LP running the CPUID instruction.
          
           NOTE:
           For a full proof solution - we have to enumerate the whole machine:
           we can pin to each CPU (using say taskset) - then run CPUID on each to collect its
           x2APIC ID - then use these shifts to slice the IDs and count distinct
           cores/packages. I could not fully pull this off during my experimentation.
           */
           int shift = eax & 0x1F;
           /* From Wiki - "the number of logical processors at the level queried is returned in EBX"
               For example, at SMT level, threads per core might be 2,
               at core level, LPs per package might be 16.
           */
           int num_logical = ebx & 0xFFFF;
           /* x2apic_id is giving x2APIC ID of whichever LP is running the code rn
               Wiki - "leaf Bh will return different values in EDX depending on which
               logical processor the CPUID instruction runs; the value returned in
               EDX is actually the x2APIC id of the logical processor."
           */
           uint32_t x2apic_id = edx;
           const char *level_names[] = {"Invalid","SMT","Core"};
           printf("  Level %d: type=%s(%d), logical_processors=%d, "
                  "bit_shift=%d, x2APIC_ID=%u\n",
                  level,
                  level_type < 3 ? level_names[level_type] : "Unknown",
                  level_type, num_logical, shift, x2apic_id);
       }
   } else {
       printf("  Not supported (max leaf < 0xB)\n");
   }

   /* Leaf 0x1F: V2 Extended Topology (newer Intel)
       Not much useful info for this blog
   */
   printf("\nLeaf 0x1F (V2 Extended Topology):\n");
   if (max_leaf >= 0x1F) {
       for (int level = 0; level < 8; level++) {
           cpuid_count(0x1F, level, &eax, &ebx, &ecx, &edx);
           int level_type = (ecx >> 8) & 0xFF;
           if (level_type == 0) break;
           int shift = eax & 0x1F;
           int num_logical = ebx & 0xFFFF;
           printf("  Level %d: type=%d, logical_processors=%d, "
                  "bit_shift=%d\n",
                  level, level_type, num_logical, shift);
       }
   } else {
       printf("  Not supported (max leaf < 0x1F)\n");
   }

   /* Leaf 0x40000000: Hypervisor Detection
       From Wiki:
       "When the CPUID instruction is executed under Intel VT-x or AMD-v virtualization,
       it will be intercepted by the hypervisor, enabling the hypervisor to return CPUID feature
       flags that differ from those of the underlying hardware.
       .....
       For leaf 40000000h, the hypervisor is expected to return the index of the highest supported
       hypervisor CPUID leaf in EAX, and a 12-character hypervisor ID string in EBX,ECX,EDX (in that order).
       "
   */
   printf("\nLeaf 0x40000000 (Hypervisor ID):\n");
   cpuid_count(0x40000000, 0, &eax, &ebx, &ecx, &edx);
   if (eax != 0 || ebx != 0 || ecx != 0 || edx != 0) {
       char hv_id[13];
       *(uint32_t*)(hv_id+0) = ebx;
       *(uint32_t*)(hv_id+4) = ecx;
       *(uint32_t*)(hv_id+8) = edx;
       hv_id[12] = '\0';
       printf("  Hypervisor ID: %s\n", hv_id);
       printf("  Max hypervisor leaf: 0x%x\n", eax);
   } else {
       printf("  All zeros (blocked by gVisor allowlist or "
              "no hypervisor leaf)\n");
   }

   return 0;
}

There is a tremendous amount of information to unpack here! "Microsoft Hv" is the Hyper-V signature. This strongly indicates that OpenAI is running their container environment on Azure. Based on this SPEC CPU result, the cache geometry itself seems to match the Ice Lake (Sunny Cove) family - Intel Xeon Gold 6338. The above CPUID output shows us 32 LPs and 2 threads per core, which means 16 physical cores per package. It is important to stress that this CPUID program only ran on one LP; therefore, it only reported the topology of its own package (32 LPs). We cannot directly tell the total number of LPs in the host, or tell how many packages exist from the above output.

One might also wonder, for this cache geometry, Intel Xeon Gold 6338 is supposed to have 32 cores per socket / 64 threads, so why are we only seeing 32 LPs? I believe the answer possibly lies with the container host allocation. The container host was handed 32 of those threads (16 cores worth). Therefore, it is not getting a whole socket, but a slice, because that is the quota the hypervisor carved out for the VM. In this case, the hypervisor does not seem to resize the cache geometry (I am not sure what the industry practice is) - which is why we see the real chip's 48 MB L3, 1.25 MB L2, and 48 KB L1d.

Note that the gVisor codepath to determine the number of cores for /proc/cpuinfo comes from tasks.go: 

func cpuInfoData(k *kernel.Kernel) string {
	features := k.FeatureSet()
	var buf bytes.Buffer
	for i, max := uint(0), k.ApplicationCores(); i < max; i++ {
		features.WriteCPUInfoTo(i, max, &buf)
	}
	return buf.String()
}

Given all that we have seen so far, there are two plausible explanations for the original 56 LP finding:

  1. OpenAI passes an explicit NumCPU argument (see loader.go above) to the gVisor sandbox.
  2. OpenAI does not pass anything explicit and falls back on runtime.NumCPU() - which would show nproc=56 when gVisor is running inside a container host that has 56 LPs in its cpuset, for instance, with two NUMA nodes. And given that the CPUID program only ran on one LP, it only reported the topology of its own package; the topology could consist of a package that contributes 32 LPs and another that contributes 24 LPs.

Across multiple re-runs of the above C code, I have not stumbled upon a 24 LP package - so I cannot say that it appears to be the case. To quantify how many actual cores of compute are available, we can ask ChatGPT to run a simple CPU parallelism benchmark as follows: 

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <pthread.h>
#include <time.h>
#include <unistd.h>

#define ITERATIONS 500000000L

static void *burn(void *arg) {
   (void)arg;
   volatile long x = 0;
   for (long i = 0; i < ITERATIONS; i++) {
       x += i;
   }
   return NULL;
}

static double now(void) {
   struct timespec ts;
   clock_gettime(CLOCK_MONOTONIC, &ts);
   return ts.tv_sec + ts.tv_nsec * 1e-9;
}

int main(void) {
   /*
       Reads from /sys/devices/system/cpu/online
       See: https://github.com/bminor/glibc/blob/master/sysdeps/unix/sysv/linux/getsysstats.c#L214
       Falls back to /proc/stat and sched_getaffinity
   */
   int max_cores = sysconf(_SC_NPROCESSORS_ONLN);

   /* Single-threaded busy loop for baseline time */
   double t0 = now();
   burn(NULL);
   double single = now() - t0;
   printf("Cores reported: %d\n", max_cores);
   printf("Single-thread:  %.2fs\n\n", single);

   /* Spawn across 1, 2, 4, ..., max_cores */
   int counts[32];
   int n = 0;
   for (int c = 1; c <= max_cores; c *= 2)
       counts[n++] = c;
   if (counts[n-1] != max_cores)
       counts[n++] = max_cores;

   for (int t = 0; t < n; t++) {
       int nthreads = counts[t];
       pthread_t *tids = malloc(nthreads * sizeof(pthread_t));

       double t0 = now();
       for (int i = 0; i < nthreads; i++)
           pthread_create(&tids[i], NULL, burn, NULL);
       for (int i = 0; i < nthreads; i++)
           pthread_join(tids[i], NULL);
       double wall = now() - t0;

       /*
       Calculates the speedup -
       if N threads each doing the same amount of work finished in the same time as single thread,
       then the parallelism is N. If it is much less than N, then we can say that we have much less
       real concurrncy than what the OS is reporting.
       */
       double effective = (single * nthreads) / wall;
       printf("%2d threads: wall=%7.2fs  effective_parallelism=%.1f\n",
              nthreads, wall, effective);

       free(tids);
   }

   return 0;
}

In the above code, we start by running a single-threaded busy loop (with 500M iterations) to establish a baseline time. Next, we spawn the same busy loop workload across 1, 2, 4, 8, … up to max threads. If N threads are each doing the same amount of work and finish in the same time as a single thread, then the parallelism is N. If it is much less than N, then we can say that we have less real concurrency than what the OS is reporting.

ChatGPT compiled the code as cc -O2 -pthread /tmp/cpu_burn.c -o /tmp/cpu_burn and produced the following output:

The headline is that the memory limit is exactly 4 GiB (4194304/1024/1024)! To understand gVisor's memory model better, I found their Resource Model guide to be a good start:

The host is able to manage physical memory using regular means (e.g. tracking working sets, reclaiming and swapping under pressure). The Sentry lazily populates host mappings for applications, and allow the host to demand-page those regions, which is critical for the functioning of those mechanisms.
The Sentry implements its own memory management, including demand-paging and a Sentry internal page cache for files that cannot be used natively. A single memfd backs all application memory. .....
Therefore, memory usage statistics within the sandbox (e.g. via proc) are approximations. The Sentry maintains an internal breakdown of memory usage, and can collect accurate information but only through a relatively expensive API call. .....

Sentry's memory management documentation describes the working in greater detail. It has this to say as to why gVisor needs memfd at all:

When the application first incurs a page fault on ..... address, the host kernel delivers information about the page fault to the sentry in a platform-dependent manner, and the sentry handles the fault ..... The sentry allocates memory to store the contents of the required part of the file, and copies file contents to the allocated memory. However, since the sentry is implemented atop a host kernel, it does not configure mappings to physical memory directly. Instead, mappable "memory" in the sentry is represented by a host file descriptor and offset, since ..... this is the memory mapping primitive provided by the host kernel. In general, memory is allocated from a temporary host file using the pgalloc package.

Intuitively, say an application wants to read a mapped file for the first time. To do this, it may need to get the file's contents from disk. However, disk IO can be slow, and many processes might be mapping the same file. For this, the Linux kernel maintains a filemap (also known as a page cache), which stores physical pages in memory holding disk contents. Now, if a new process wants to map a file that is already mapped, it can reuse the same physical pages. To do this, the kernel creates a Page Table Entry (PTE) in the new process' page tables pointing to the already mapped page frame. Unfortunately, gVisor cannot configure mappings to physical memory directly because it is not a kernel; it is a mere regular program running in userspace.

To mitigate this limitation, Sentry creates and manages a memory mapped file (memfd). This ties back to what the Resource Model guide mentioned - "A single memfd backs all application memory". When an application running in a gVisor environment needs memory, the Sentry allocates an offset from this memfd and asks the host to map it into the application's address space using mmap. The host then handles allocating physical memory and installing page table entries.

So how does this connect to the 4 GiB memory number? My understanding is that the container runtime sets the memory limit - something like "docker run" with --memory=4g. During sandbox creation, gVisor's runsc reads this limit and passes it as a --total-memory argument during boot time. From runsc/cmd/boot.go: 

	f.Uint64Var(&b.totalMem, "total-memory", 0, "sets the initial amount of total memory to report back to the container")

This value then flows into runsc/boot/loader.go, where both the minimum and maximum memory are pinned to the same number: 

if args.TotalMem > 0 {
		// Adjust the total memory returned by the Sentry so that applications that
		// use /proc/meminfo can make allocations based on this limit.
		usage.MinimumTotalMemoryBytes = args.TotalMem
		usage.MaximumTotalMemoryBytes = args.TotalMem
		log.Infof("Setting total memory to %.2f GB", float64(args.TotalMem)/(1<<30))
	}

This is the ceiling. Before allocating from memfd, gVisor's memory allocator checks against this limit to deal with potential memory pressure. The entire /proc/meminfo output itself is constructed by a Generate() call in pkg/sentry/fsimpl/proc/tasks_files.go:

So, can we push the ChatGPT container to its memory limit? Yes, we certainly can! Here is a simple benchmarking program I worked on with Claude for this: 

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

#define CHUNK_SIZE (64 * 1024 * 1024)

int main(void) {
    size_t total = 0;
    int chunk_num = 0;

    while (1) {
        void *p = malloc(CHUNK_SIZE);
        if (!p) {
            printf("malloc failed after %zu MB\n", total / (1024 * 1024));
            break;
        }
        memset(p, 0xAA, CHUNK_SIZE);
        total += CHUNK_SIZE;
        chunk_num++;
        printf("Allocated chunk %d: %zu MB total\n", chunk_num, total / (1024 * 1024));
    }

    return 0;
}

In the above code, we keep requesting a chunk of 64 MiB memory in an infinite loop. Then, we invoke memset, filling every byte of the chunk with 0xAA. This forces the pages to be faulted, which triggers demand paging, and real physical memory gets committed to it. So, in summary, ChatGPT containers have a pretty nifty compute capacity for a $20 price point - ~4 effective LPs and 4 GB total RAM (~3.5 GB available after sandbox overhead). One could make a strawman argument here saying that a 2 LP and 4 GB RAM VPS on Linode costs $0.07/hr. However, an important point to note is that their container environment design philosophy seems to mimic a "bursty-traffic" pattern. They expect a significant delay between two successive container runs per user per session - as it is likely that the user will pause and evaluate the previous runtime response. This can be seconds or even a few minutes. It helps them reallocate their compute capacity elsewhere, reducing the stress on their system. Not to mention the fact that according to their own report "How People Use ChatGPT" from September 2025 - only 5-8% of their users use ChatGPT for technical help (3% mathematical calculation, 0.4% data analysis, and 4.2% programming). This includes people who use it for common use cases such as code writing, technical information, and getting debugging steps. OpenAI's free non-login account does not have a container environment; however, Simon Willison did mention that their free sign-in account briefly showed the container instance working.

Purely out of fun, I performed some napkin math to calculate the compute required to support their "paid" users - however, I would strongly take these numbers with a grain of salt. Wild wild west territory ahead!

As of June 6, 2026, ChatGPT has around 50 million paid subscribers. OpenAI's paper reports that ChatGPT saw 18B messages/week from 700M weekly users in July 2025. That would be 18B/700M/7days or 3.67 messages per weekly active user per day. With around 50 million paying users, a technical help share of 5-8%, and assuming that 20% of those technical help messages require a fresh container start, the math for container starts per day would be 3.67 * (5 to 8)/100 * 50 * 20/100 or 1.84M to 2.93M container starts per day. Assuming a 5-minute average container lifetime, that implies about (1.84M to 2.93M)/86400 * 5 * 60 or 6,388 to 10,173 concurrent container environments. Even if the technical users are 2x as active, the estimate rises to about ~20k concurrent container environments.

The advent of local tool-calling and agentic CLI tools like Claude Code might have prevented the use of web-based containers from exploding too.
